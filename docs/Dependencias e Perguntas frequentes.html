1.1.1 BCrypt.Net-Next

BCrypt.Net-Next v4.1.0 Int1
Hash one-way de senhas (BCrypt) com salt automático e fator de custo ajustável. Padrão da indústria pra armazenar senhas com segurança.

Por que usamos: precisamos guardar a senha do administrador e a senha de cada pessoa cadastrada de forma irreversível. Mesmo se alguém roubar o banco, a senha original não é recuperável — só dá pra verificar se uma senha digitada bate com o hash. BCrypt resiste a ataques de força bruta porque cada hash leva centenas de milissegundos pra calcular (configurável via "fator de custo").

Funções utilizadas:
  • BCrypt.Net.BCrypt.HashPassword(senha, workFactor: 10) — gera o hash com fator 10. Usado no cadastro de pessoa (Pessoas/Index.razor → HandleCadastrar) e ao trocar senha de admin.
  • BCrypt.Net.BCrypt.Verify(senhaInformada, admin.senhaHash) — confere se a senha digitada bate com o hash armazenado. Usado em AuthController.Login e no autenticador de acesso por senha.

1.1.2 Hangfire (Core, AspNetCore, MemoryStorage)

Hangfire v1.8.14 Int1
Framework de jobs em background pra .NET. Permite agendar tarefas recorrentes (cron) ou disparar tarefas em segundo plano sem bloquear a requisição HTTP. Tem dashboard web pra acompanhar o status.

Por que usamos: precisamos rodar duas tarefas de limpeza todo dia às 03:00 UTC sem interferir no funcionamento normal: (1) apagar tentativas de acesso e logs com data de retenção vencida; (2) inativar pessoas que ficaram tempo demais sem acessar. Hangfire faz isso de forma robusta, com retry automático em caso de falha e visibilidade total no dashboard.

Funções utilizadas:
  • builder.Services.AddHangfire(...) e AddHangfireServer() — registra o Hangfire no startup do Int1 (Banco/.../Program.cs).
  • RecurringJob.AddOrUpdate<InativarUsuariosInativos2AnosJob>("inativar-usuarios", j => j.Executar(), "0 3 * * *") — agenda o job de inativação pras 03:00 UTC diariamente.
  • RecurringJob.AddOrUpdate<LimparDadosExpiradosJob>("limpar-dados", j => j.Executar(), "0 3 * * *") — agenda o job de purga.
  • app.UseHangfireDashboard("/hangfire") — expõe o dashboard web em http://localhost:5018/hangfire pra monitorar jobs.

1.1.3 Microsoft.EntityFrameworkCore.Sqlite (+ Design, Tools)

Entity Framework Core + SQLite Provider v8.0.7 Int1, Int2 (transitivo), Int3 (transitivo)
ORM oficial da Microsoft pra .NET. Em vez de escrever SQL na mão, você define classes C# (entidades) e o EF traduz pra comandos SQL. O provider SQLite faz tudo isso em cima de um arquivo único de banco (sem servidor).

Por que usamos: precisamos armazenar pessoas, ambientes, T50Ms, câmeras, tentativas de acesso e logs. SQLite foi escolhido em vez de SQL Server/PostgreSQL porque: (1) o sistema roda numa máquina só, sem necessidade de banco distribuído; (2) zero administração — é um arquivo banco.db que dá pra backup com simples cp; (3) altamente performático pra cargas moderadas (milhares de acessos por dia). EF Core dá produtividade enorme: migrações automáticas, queries via LINQ, change tracking.

Funções utilizadas:
  • builder.Services.AddDbContext<AppDbContext>(opt => opt.UseSqlite(...)) — registra o contexto do banco no startup.
  • db.Pessoas.Where(...).Include(p => p.Ambientes).ToList() — queries com eager loading (necessário porque EF Core não tem lazy loading por padrão).
  • db.Pessoas.Add(pessoa), db.SaveChangesAsync() — inserts/updates.
  • db.Database.EnsureCreated() — cria o banco se não existir (usado no startup).
  • Migrations via CLI: dotnet ef migrations add <nome> e dotnet ef database update.

1.1.4 Microsoft.AspNetCore.Authentication.JwtBearer + System.IdentityModel.Tokens.Jwt

JWT Bearer Authentication v8.0.0 / v7.6.0 Int1
Implementação padrão de autenticação por tokens JSON Web Token (JWT). O cliente loga uma vez, recebe um token assinado, e nas próximas requisições envia o token no header Authorization: Bearer ....

Por que usamos: qualquer endpoint da API precisa ser autenticado, exceto o de login. JWT é stateless (não precisa guardar sessão no servidor), expira sozinho em 8h, e permite carregar dados do admin diretamente no token (adminId, nomeCompleto) sem precisar consultar o banco a cada requisição.

Funções utilizadas:
  • builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme).AddJwtBearer(...) — configura JWT Bearer no startup.
  • new JwtSecurityToken(issuer, audience, claims, expires: DateTime.UtcNow.AddHours(8), signingCredentials: ...) — gera o token no AuthController.Login.
  • new JwtSecurityTokenHandler().WriteToken(token) — serializa o token pra string.
  • [Authorize] nos controllers — exige token válido.

1.1.9 Anviz.SDK

Anviz.SDK v2.0.18 Int2 (Worker)
SDK não-oficial mas amplamente utilizado pra falar com dispositivos da família T-series da Anviz (T5, T50, T50M) via protocolo OAE (Open Anviz Ethernet) sobre TCP/IP. Wraps o protocolo binário em chamadas .NET assíncronas.

Por que usamos: escrever do zero o protocolo OAE seria enorme (centenas de comandos, framing/checksum, criptografia). O SDK encapsula tudo isso em métodos como SetEmployeesData(), EnrollFingerprint(). Detalhes técnicos do T50M estão na §1.2.

Funções utilizadas em AnvizConnector.cs e AnvizService.cs:
  • new AnvizManager().Connect(ip, porta).Result — estabelece sessão TCP com o T50M e retorna um AnvizDevice.
  • _device.ReceivedRecord += OnReceivedRecord — assina o evento de novo registro de acesso (push do T50M).
  • _device.DownloadRecords(onlyNew: true).Result — drena registros offline acumulados no T50M durante queda de rede.
  • _device.SetEmployeesData(userInfo).Wait() — cadastra/atualiza um usuário no T50M.
  • _device.DeleteEmployeesData(id).Wait() — remove usuário.
  • _device.SetFingerprintTemplate(id, finger, template).Wait() — envia template biométrico já capturado anteriormente (usado pra restaurar biometria).
  • _device.GetFingerprintTemplate(id, finger).Result — baixa template biométrico do T50M (backup).
  • _device.EnrollFingerprint(id).Result — pede ao T50M pra capturar nova digital. Bloqueia até o usuário colocar o dedo 2 vezes.
  • _device.SetDateTime(DateTime.Now).Wait() — sincroniza relógio do T50M com o servidor.

1.2 Dispositivo T50M Anviz

Que garantia eu tenho que o T50M vai funcionar?

A garantia vem em 3 camadas:

  1. Protocolo oficial Anviz: o T50M usa OAE (Open Anviz Ethernet), publicado pela fabricante e estável há mais de uma década. A SDK Anviz.SDK 2.0.18 implementa esse protocolo e é a mesma usada por sistemas comerciais (Anviz CrossChex etc).
  2. O T50M é um dispositivo padrão de mercado: não é hardware customizado nosso. Qualquer T50M comprado da Anviz vai responder ao protocolo OAE da mesma forma.

Ressalva conhecida: existe um ponto a investigar caso autenticação por senha falhe — o T50M usa um formato especial de 3 bytes pra senha numérica, e a SDK .NET pode ou não fazer essa conversão internamente. Se houver problema, está documentado no código em AnvizService.cs linhas 22-26 onde olhar primeiro.

Como o nosso sistema conecta com o T50M?

A conexão é TCP/IP direta na porta 5010 com protocolo binário OAE. O fluxo é:

  1. O T50M é configurado com um IP fixo na rede local (pelo menu do próprio aparelho: System → Comm → Ethernet).
  2. O admin cadastra esse IP no painel (Dispositivos T50 → + Novo T50).
  3. O Worker (Int2) lê do banco a lista de T50Ms cadastrados e, pra cada um, instancia um AnvizConnector com aquele IP e porta.
  4. O AnvizConnector.Conectar() chama new AnvizManager().Connect(_ip, _porta).Result — isso abre o socket TCP e estabelece a sessão OAE com o dispositivo.
  5. Uma vez conectado, o Worker assina o evento _device.ReceivedRecord — sempre que alguém usa o T50M (passa o dedo ou digita senha), o T50M envia um Record push pro Worker.
  6. O Worker processa o evento (EventProcessor.Processar) e registra a tentativa no banco.

O que o cliente tem que fazer (ou mudar no código) pra funcionar?

Não precisa mudar nada no código. Tudo é configuração:

  1. Configurar IP fixo em cada T50M (menu local do aparelho).
  2. Garantir que o servidor da aplicação enxerga os T50Ms via TCP/IP na porta 5010.
  3. Cadastrar cada T50M no painel (com nome descritivo e o IP).
  4. (Opcional) Definir o modo de verificação local no T50M (FP+PWD recomendado).

O único cenário em que seria preciso mexer no código é se a senha de 3 bytes falhar autenticação: aí seria preciso adicionar a conversão manual em AnvizService.cs:AdicionarPessoa(), linha 26 (userInfo.Password = ulong.Parse(senha)) — substituir por uma conversão pro formato Anviz se necessário.

1.3 Câmeras IP (ONVIF / RTSP / HLS)

Como o nosso sistema conecta com as câmeras?

Não há "conexão persistente" com a câmera — o sistema fala com ela sob demanda, em 2 momentos:

  1. Quando o admin pede pra ver ao vivo: o painel apenas constrói uma URL RTSP/HLS e a entrega pro player do navegador. A câmera é falada pelo navegador, não pelo nosso backend.
  2. Quando uma tentativa de acesso é registrada: o CameraService.MonitorarNovoArquivo() espera o tempo configurado (30-120s), conecta no endpoint ONVIF da câmera via SimpleOnvifClient, faz um GetDeviceInformationAsync() pra confirmar que ela está respondendo, e monta uma URL RTSP de Replay no formato Hikvision/Dahua/Intelbras (rtsp://user:pass@ip/Streaming/tracks/101?starttime=...&endtime=...). Essa URL é salva no campo GravacaoPath da tentativa.

É um modelo deliberadamente leve: o sistema não baixa, transcodifica ou armazena gravações. A câmera continua gravando como já fazia. Nosso sistema só guarda o link pra encontrar o trecho certo.

Com que tipo de câmeras o sistema funciona?

Qualquer câmera IP que suporte ONVIF Profile S e disponibilize um stream RTSP. Isso cobre praticamente todas as marcas profissionais:

  • Intelbras (séries VIP, série S)
  • Hikvision (DS-2CD..., Hilook)
  • Dahua (IPC-HFW..., IPC-HDW...)
  • Axis (M, P, Q series)
  • Bosch (Dinion, Flexidome)

Câmeras analógicas convencionais (sem rede) não funcionam — o sistema requer câmera IP com ONVIF + RTSP. Câmeras Wi-Fi residenciais (tipo Wyze, Tapo) geralmente não têm ONVIF aberto e não funcionam direto; em alguns casos firmwares alternativos (CFW) habilitam.

O que é a URL RTSP e pra que serve?

RTSP (Real Time Streaming Protocol) é o protocolo padrão pra entrega de vídeo ao vivo de câmeras IP. A URL tem o formato:

rtsp://<usuario>:<senha>@<ip-da-camera>:554/<caminho-do-stream>

Exemplo real Hikvision:

rtsp://admin:S3nh@F0rt3@192.168.0.50:554/Streaming/Channels/101

Pra que serve no sistema:

  • É a URL que o admin pode abrir no VLC pra ver a câmera ao vivo (player desktop).
  • É a base usada pra construir a URL de Replay que aparece no Histórico — o sistema só substitui o path por /Streaming/tracks/101?starttime=...&endtime=... e usa as mesmas credenciais.
  • É obrigatória no cadastro.
O que é a URL HLS e pra que serve? Por que é opcional?

HLS (HTTP Live Streaming) é o protocolo de streaming sobre HTTP usado pra rodar vídeo em navegadores web modernos. Funciona em qualquer navegador (Chrome, Edge, Firefox) sem precisar de plugin.

Exemplo de URL HLS:

http://mediamtx.local:8888/cam-frontal/index.m3u8

Por que é opcional: as câmeras IP não disponibilizam HLS diretamente — elas só falam RTSP. Pra ter HLS, é necessário um servidor de mídia intermediário (FFmpeg, nginx-rtmp ou MediaMTX) que pega o RTSP da câmera e retransmite como HLS pro navegador. Como nem todo cliente vai querer instalar esse intermediário, deixamos o campo opcional.

Se HLS for preenchido: o painel mostra o vídeo ao vivo direto na aba "Ver ao Vivo".

Se HLS for omitido: a aba "Ver ao Vivo" mostra um placeholder ("vídeo indisponível no navegador, use VLC com a URL RTSP"). O resto do sistema (gravação associada, etc.) continua funcionando normalmente.

O que é o Endereço ONVIF e pra que serve? É obrigatório?

ONVIF (Open Network Video Interface Forum) é um padrão aberto que permite que sistemas de diferentes fornecedores conversem com câmeras de diferentes marcas. Toda câmera IP profissional implementa ONVIF.

O endpoint ONVIF é uma URL SOAP/HTTP pra falar com a câmera. Formato típico:

http://<ip-da-camera>:80/onvif/device_service

Pra que serve no nosso sistema:

  • Quando ocorre um acesso liberado, o CameraService conecta no endpoint ONVIF da câmera do ambiente e chama GetDeviceInformationAsync() — isso confirma que a câmera está respondendo e que as credenciais embutidas na URL RTSP são válidas.
  • Só depois desse "ping ONVIF" bem-sucedido é que o sistema monta a URL de Replay e associa ao histórico.

É obrigatório?

  • Pra ter gravação associada no histórico: sim. Sem ONVIF, o campo "Vídeo" no histórico fica em branco (—).
  • Pra ver ao vivo: não. Ver ao vivo funciona só com a URL RTSP (via VLC) ou HLS (via navegador).

O campo no formulário aceita ficar vazio — você cadastra a câmera só com RTSP se não tem interesse em rastrear gravações por acesso.

Qual a limitação do sistema em relação às câmeras?
  • Não controlamos a gravação: o sistema assume que a câmera já está gravando (por detecção de movimento ou contínuo) — só pega o link do trecho. Se a câmera não estiver gravando, não há vídeo.
  • O Profile G completo do ONVIF não está implementado. A SDK SharpOnvifClient na versão usada não expõe FindRecordings + GetReplayUri de forma confiável pra todas as marcas. Por isso usamos a abordagem "montar URL no formato Hikvision/Dahua/Intelbras", que funciona pra essas 3 marcas (e similares) mas pode não funcionar pra outras marcas.
  • Streaming ao vivo no navegador exige HLS: RTSP não roda em navegador moderno. Se quiser ver ao vivo direto no painel, precisa do MediaMTX configurado.
  • Sem PTZ: o sistema não controla pan/tilt/zoom da câmera. Visualização é só estática.
  • Sem áudio: não consumimos nem armazenamos áudio das câmeras.
Que garantia temos que vai funcionar?
  • O protocolo ONVIF é padronizado pela ONVIF Foundation desde 2008. Câmeras certificadas são obrigadas a responder ao GetDeviceInformation de forma consistente.
  • O formato de URL de Replay /Streaming/tracks/101?starttime=...&endtime=... é o padrão Hikvision e foi adotado também por Dahua e Intelbras (que copiaram esse formato pra interoperar com NVRs da concorrência).
  • Validamos com VLC: antes do cadastro, qualquer câmera que abra no VLC vai funcionar pro ver ao vivo. Se também responde ONVIF (testável com programas como ONVIF Device Manager), funciona pra gravação associada.

Ressalva: não dá pra garantir 100% sem testar com a câmera específica do cliente. O comportamento esperado:

  • Câmera responde ONVIF + tem URL RTSP no formato padrão → tudo funciona.
  • Câmera responde ONVIF mas formato de Replay diferente (raro) → ver ao vivo OK, gravação associada vai dar 404 ao tentar abrir.
  • Câmera não responde ONVIF → ver ao vivo OK (RTSP funciona), gravação associada não vem (— no histórico).
O que tem que fazer (ou mudar no código) pra funcionar?

Não precisa mexer no código. Só configuração na câmera:

  1. Habilitar ONVIF no painel web da câmera (caminho varia por marca).
  2. Criar um usuário ONVIF dedicado (não usar o admin da câmera) com permissão de leitura + playback.
  3. Anotar URL RTSP, endpoint ONVIF e credenciais.
  4. Cadastrar tudo no painel do sistema.

Se quiser ver ao vivo no navegador: instalar e configurar MediaMTX (ou FFmpeg/nginx-rtmp), apontar pra URL RTSP da câmera, copiar a URL HLS gerada pro campo HLS do cadastro. Setup detalhado está no DeploymentGuide.html, §6.4.

1.4 Envio de email

Como o nosso sistema está fazendo o envio de email?

O fluxo completo, passo a passo:

  1. Quando o admin clica em "Cadastrar" no formulário de pessoa, depois da gravação no banco, o Frontend (Int3) chama o endpoint POST /api/person/{id}/reenviar-credenciais do Int1.
  2. O PersonController.ReenviarCredenciais() faz:
    1. Busca a pessoa no banco.
    2. Descriptografa a senhaClear usando AES-256-CBC com a chave do appsettings.
    3. Lê as 4 variáveis de ambiente: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS.
    4. Se alguma estiver faltando: cai no fallback console — escreve no log do servidor as credenciais e marca fallback = true no retorno.
    5. Se todas estiverem presentes: monta um MimeMessage com HTML, conecta no SMTP via SmtpClient.ConnectAsync(host, port, SecureSocketOptions.StartTls), autentica e envia.
    6. Registra log de auditoria (LogAdmin com ação "ReenvioCredenciais").
    7. Retorna JSON: { sucesso: true, fallback: true/false, mensagem: "..." }.
  3. O Frontend usa esse retorno pra mostrar no painel: verde (enviado), amarelo (fallback — SMTP falhou), azul (enviando).

O HTML do email mora em PersonController.cs linhas 182-189, na variável corpoHtml — texto HTML simples com tabela do ID e senha. Pra customizar (cores, logo, mensagem), basta editar essa string e rebuildar.

Que garantia temos que vai funcionar?

A garantia tem 2 partes:

Do lado do nosso código: a biblioteca MailKit é a referência de mercado pra clientes SMTP em .NET, mantida ativamente e usada em centenas de milhares de aplicações. O código segue exatamente o template oficial da documentação MailKit. Não há "mágica" nem heurística — é SMTP padrão (RFC 5321).

Do lado do servidor SMTP do cliente: isso depende exclusivamente do servidor deles. Se o EBmail Zimbra estiver online, aceitar conexões TLS em 587 e as credenciais funcionarem em qualquer outro cliente de email (Outlook, Thunderbird), também vai funcionar no nosso sistema.

Como confirmar funcionamento antes da entrega definitiva: no painel, ao cadastrar uma pessoa de teste com o email do próprio admin, a banner de status já mostra se chegou. Se não chegar, o log do Int1 indica exatamente o erro (auth failed, timeout, TLS failed, etc.). Veja a seção 7.3 do DeploymentGuide.html.

Vai enviar pelo servidor de email ? Como?

Sim, integralmente pelo servidor SMTP deles. O sistema é apenas um cliente SMTP — não tem servidor de email embutido. Não faz relay próprio nem usa serviço externo (SendGrid, etc.).

A configuração funciona por variáveis de ambiente lidas pelo processo do Int1 no startup:

VariávelValor exemploSignificado
SMTP_HOSTsmtp.ebmail.eb.mil.brHostname do servidor SMTP do CTA
SMTP_PORT587Porta SMTP — padrão STARTTLS é 587, SSL puro é 465
SMTP_USERcontrole.acesso@eb.mil.brConta de serviço dedicada pra envio
SMTP_PASS(privado)Senha SMTP da conta

Quando o Int1 sobe, ele lê essas 4 variáveis com Environment.GetEnvironmentVariable("SMTP_HOST") etc. e usa nas chamadas MailKit. Se alguma estiver vazia, cai no fallback console.

O que o tem que fazer (ou mudar no código) pra funcionar?

Não precisa mexer no código. Só configuração de ambiente, em 3 passos:

  1. Obter da TI do CTA: SMTP_HOST, SMTP_PORT, SMTP_USER (idealmente uma conta dedicada do tipo controle.acesso@eb.mil.br), SMTP_PASS.
  2. Configurar as 4 variáveis no nível da máquina (não da sessão), de forma persistente: 
    # Windows (PowerShell como Admin)
[System.Environment]::SetEnvironmentVariable("SMTP_HOST", "smtp.ebmail.eb.mil.br", "Machine")
[System.Environment]::SetEnvironmentVariable("SMTP_PORT", "587", "Machine")
[System.Environment]::SetEnvironmentVariable("SMTP_USER", "controle.acesso@eb.mil.br", "Machine")
[System.Environment]::SetEnvironmentVariable("SMTP_PASS", "<senha>", "Machine")
  3. Reiniciar o Int1 (as env vars são lidas no startup): Restart-Service "5CTA-Banco" ou rodar o iniciar.ps1 de novo.

No Linux, as 4 variáveis vão no arquivo do serviço systemd em /etc/systemd/system/5cta-banco.service. Detalhes na §7.2 do DeploymentGuide.html.

Customização do conteúdo do email (opcional): se quiser mudar o HTML do email (adicionar logo do 5º CTA, mudar cores, etc.), editar a variável corpoHtml em PersonController.cs linhas 182-189 e rebuildar.
